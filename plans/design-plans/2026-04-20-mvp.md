# Friski MVP (Phase 0) Design

## Summary

Friski is a structured, claim-based local news wiki for San Francisco, built on a three-repository architecture: `friski-code` (the Astro 5 application, schemas, scripts, and CI), `frisco-wiki` (the content repository of article prose and subject YAML files), and a deferred archival store. The two active repos are joined at build time via a git submodule: Netlify checks out `friski-code` with `frisco-wiki` pinned inside it, constructs an in-memory subject graph from the YAML corpus, and pre-renders a fully static site with no runtime database.

The central design bet is that factual claims are first-class structured data, not prose. Each subject (a person, institution, or place) carries a list of claims in Wikidata vocabulary, each claim pinned to a versioned archived source. Articles are prose views that reference subjects and resolve footnote citations against those source records. A GitHub Actions reviewer — using the Claude API — checks incoming content PRs for three things: whether prose assertions are backed by claims, whether cited sources actually support those assertions, and whether the prose meets NPOV standards. This split between a thin content repository with an opinionated claim schema and a code repository that owns all validation, rendering, and review logic is the shape that all later phases extend.

## Definition of Done

The MVP is complete when a full round-trip works on 2–3 stub articles:

1. A new subject is scaffolded via `scripts/new-subject.ts` from a Wikidata QID, producing a valid subject YAML with minimal P31 + other claims.
2. Source URLs for that subject are submitted to Wayback via `scripts/ensure-archived.ts`; resulting snapshot URLs are written back into the subject file.
3. A prose article referencing the subject is authored in `frisco-wiki`, using terse footnote citations (`[^label]: source-id`).
4. A PR to `frisco-wiki` passes `scripts/validate-content.ts` (schema + cross-reference checks) and triggers the reviewer action.
5. The reviewer produces a PR comment with findings from three checks (claim coverage, source support, NPOV). Findings are judged by Luis to be *actionable and honest* — not noise, not flattery — including on at least one deliberately-flawed stub designed to exercise each check.
6. Merge to `frisco-wiki` main triggers an automated submodule-bump PR against `friski-code`. Merging that PR triggers a Netlify build.
7. `frisco.wiki` renders the new article, its subject page, and any affected cross-subject index pages.
8. Adding a subsequent subject + article takes minutes, not an expedition.

MVP is explicitly **not** about breadth of content (2–3 stubs are enough) or polish of editorial experience. It is about proving the workflow machinery works end-to-end on content that would pass a competent reader's smell test.

## Glossary

- **Wikidata QID**: A globally unique identifier for an entity in the Wikidata knowledge base, formatted as `Q` followed by a number (e.g., `Q99524088`). Used in Friski to seed a subject file and to borrow property and value vocabulary.
- **P-number / P31**: Wikidata's notation for a property identifier. `P31` specifically means "instance of" — the property that classifies what kind of thing a subject is (e.g., `P31: Q5` means "instance of human"). Friski reuses these identifiers where possible to avoid inventing parallel vocabulary.
- **Q-number**: Wikidata's notation for an entity identifier used as a claim value (e.g., `Q5` = human). In Friski, Q-numbers appear as claim values when the referent has a Wikidata entry.
- **F- prefix**: Friski-local identifier namespace for entities that exist in the wiki but have no Wikidata entry (e.g., `F-SF-D9-Supervisor`). Keeps local coinages visually distinct from borrowed Wikidata vocabulary.
- **Subject**: Friski's unit of structured data — a person, institution, place, or event described by a YAML file containing claims and sources. Articles are prose views over one or more subjects; subjects are not articles.
- **Claim**: A single structured assertion about a subject, carrying a property, value, optional date range, and exactly one source. The atomic unit of Friski's knowledge representation.
- **Subject graph**: The in-memory data structure built at Astro build time from all subject YAML files — subjects keyed by ID, claims indexed by property, sources resolved, and derived predicates computed. Pages are rendered by iterating or filtering this graph.
- **Astro 5**: A static-site framework with a content-layer API. Used here to load, validate, and render the subject graph into a pre-rendered static site without a runtime database.
- **Zod**: A TypeScript schema validation library. Used to define and enforce the shapes of subject and article YAML files at both build time and in the standalone validator script.
- **Content-layer loader**: Astro's mechanism for ingesting structured content from arbitrary sources. In Friski, a custom loader reads `subjects/*.yaml` and `articles/*.md` from the `frisco-wiki` submodule.
- **Submodule / submodule-bump**: A git submodule pins one repository inside another at a specific commit. The submodule-bump action automates opening a PR against `friski-code` whenever `frisco-wiki` main advances, keeping the deployed site in sync.
- **Wayback / Save Page Now**: The Internet Archive's Wayback Machine and its API for submitting URLs for capture. `ensure-archived.ts` uses this API to create permanent snapshots of source URLs and writes the resulting `web.archive.org` URLs back into subject files.
- **Tier**: A numeric source-quality classification (`1`–`4`) on each source record. Not further defined in this document; the schema enforces that a value is present and in range.
- **NPOV**: Neutral Point of View — an editorial standard requiring prose to represent facts and disputes without advocacy, loaded language, or synthetic consensus. Borrowed from Wikipedia editorial tradition; Friski's specific interpretation lives in `docs/editorial-principles.md`.
- **BLP**: Biographies of Living Persons — an editorial-policy category (also from Wikipedia tradition) calling for heightened sourcing standards when making claims about living individuals. In Friski, BLP applicability is derived from the claim graph (`P31 ⊇ Q5` and no death date `P570`) rather than tagged explicitly.
- **Claim coverage check**: The reviewer step that compares article prose assertions against the subject's structured claims to detect assertions with no backing claim (or claims that don't reach as far as the prose implies).
- **Source support check**: The reviewer step that fetches archived source text and checks whether it actually supports the prose assertion citing it — an external, network-dependent check distinct from claim coverage.
- **`pull_request` vs `pull_request_target`**: Two GitHub Actions event triggers with different security postures. `pull_request` runs in the fork's context (no access to repo secrets); `pull_request_target` runs in the base repo's context (has secrets, but must not check out untrusted PR code). The document defers the migration until external contributors arrive.
- **`new-subject.ts`**: The authoring script that bootstraps a subject YAML file from a Wikidata QID, fetching label, description, and a handful of seed properties. The resulting file is authoritative after generation — the script is a one-time scaffold, not a live sync.
- **`ensure-archived.ts`**: The authoring script that submits source URLs to Wayback's Save Page Now API and writes the resulting snapshot URLs back into subject files. Run locally before committing, not in CI at MVP.
- **`validate-content.ts`**: The standalone validator script that enforces schema conformance, cross-reference integrity, and footnote resolution rules on the content corpus. Runs in CI in under 5 seconds, independently of the Astro build.
- **Allowlist (`allowed-types.yaml`)**: A configuration file enumerating the P31 values (subject types) that the site renders correctly. Additions are explicit; unknown P31 values cause a validator failure rather than silent misbehavior.
- **`frisco-wiki`**: The content-only repository holding article Markdown files and subject YAML files. Deliberately thin — no build tooling, no application code.
- **`friski-code`**: The application repository holding the Astro site, Zod schemas, authoring scripts, validator, reviewer, and CI workflows.

## Architecture

Three git repositories structure the MVP:

| Repo | Visibility | Role |
|---|---|---|
| `friski-code` | public | Astro 5 application, Zod content schemas, authoring scripts, validator, reviewer, CI workflows |
| `frisco-wiki` | public | Article prose (`articles/*.md`) and subject claims (`subjects/*.yaml`). Nothing else. |
| `frisco-archives` | — | **Not built at MVP.** Deferred to Phase 1. See `plans/deferred/archival-and-captures.md`. |

`friski-code` embeds `frisco-wiki` as a git submodule at `src/content/wiki/`. Netlify builds the pinned submodule revision. A small automated submodule-bump action opens a PR against `friski-code` whenever `frisco-wiki` main advances, keeping the deployed corpus in sync without manual follow-up.

### Data model

Subjects own claims; articles reference subjects. Sources are scoped to the subject that cites them (duplication across subjects is tolerated at MVP — no shared registry yet). Article frontmatter never restates claim data.

**Subject shape** (`subjects/<slug>.yaml`):

```yaml
id: jackie-fielder
wikidata_qid: Q99524088
label: Jackie Fielder
description: Member of the SF Board of Supervisors representing District 9.
scope: [person, politician, district-9]

claims:
  - id: C000
    property: P31                     # instance of (Wikidata P-number)
    value: Q5                         # human (Wikidata Q-number)
    source: wd-jackie-fielder
  - id: C001
    property: P39                     # position held
    value: F-SF-D9-Supervisor         # F- prefix = Friski-local identifier
    start: 2025-01-08
    end: null
    source: missionlocal-2024-11-fielder-elected

sources:
  - id: missionlocal-2024-11-fielder-elected
    url: https://missionlocal.org/...
    publication: Mission Local
    author: J. Reporter
    date_published: 2024-11-07
    tier: 1
    archive:
      url: https://web.archive.org/web/2024/...
      method: wayback
```

Claims carry exactly `{id, property, value, start?, end?, source}` — nothing more. No `type` field, no `editorial_class`. Policies that could be carried as claim-level tags (BLP handling, contested-topic treatment) are instead **derived from the claim graph**. For example: `is_living_person(subject) ⟺ P31 ⊇ {Q5} ∧ ¬∃ P570`. If a policy cannot be derived from existing claims, the right answer is usually to add a real claim property for it, not a sideband field.

Property identifiers (P-numbers) and subject/value identifiers (Q-numbers) reuse Wikidata's vocabulary where possible. Friski-local identifiers use an `F-` prefix. An allowlist file (`config/allowed-types.yaml`) constrains which P31 values render correctly; additions to the list are explicit, not silent.

**Article shape** (`articles/<slug>.md`):

```markdown
---
title: Jackie Fielder
slug: jackie-fielder
primary_subject: jackie-fielder
subjects: [jackie-fielder, sf-board-of-supervisors]
scope: [person, politician]
tags: [district-9]
---

Jackie Fielder is a member of the San Francisco Board of Supervisors,
representing District 9 since January 2025.[^fielder-elected]

[^fielder-elected]: missionlocal-2024-11-fielder-elected
```

Prose cites sources via standard Markdown footnotes. The footnote body is a source ID resolved against the sources declared by any subject in `subjects[]`. When a source ID is unique across those subjects, resolution is automatic. When ambiguous, the validator fails with a message requiring the explicit `<subject>/<source>` form. Resolution is always deterministic; no fuzzy matching.

### Reviewer check chain

```
prose assertion  ←[coverage]→  claim  ←[schema]→  source record  ←[support]→  source text
                   internal                                          external
```

- **Claim coverage** (internal): is the prose assertion backed by a claim on some referenced subject?
- **Source support** (external): does the cited source's actual text support the assertion?

Four failure modes the reviewer distinguishes:

| Scenario | Coverage | Support | Meaning |
|---|---|---|---|
| Healthy | ✓ | ✓ | nothing wrong |
| Missing claim — prose asserts something the source supports but no claim exists | ✗ | ✓ | claim should be added to the subject |
| Overreach — prose says more than the source supports | ✓ | ✗ | tighten prose or find a better source |
| Invented assertion — no claim, no source support | ✗ | ✗ | remove or re-source |
| Bad sourcing — claim matches prose, but the claim's source doesn't actually support it | ✓ | ✗ | re-source the claim |

### Build pipeline

1. Netlify checks out `friski-code` with its `frisco-wiki` submodule.
2. Astro's content-layer loader reads `subjects/*.yaml` and `articles/*.md`, validates each with Zod, builds an in-memory **subject graph** (subjects → claims → sources, with derived predicates) and **article index** (articles → resolved footnotes).
3. Pages pre-render from the graph: article pages, subject pages, and index pages derived from filtering the claim graph (e.g., "current D9 supervisors").
4. Static site ships to `dist/`; Netlify serves `frisco.wiki`.

No SQLite, no sql.js, no runtime database. Query pages are pre-rendered at build time by iterating the in-memory graph. A `dist/subjects.json` export can be emitted if an external tool ever needs the graph, but it is not part of MVP.

### Validator

`scripts/validate-content.ts` runs independently of Astro's build — fast (<5s) and in CI. It enforces:

- Zod schema conformance per subject and article.
- `subject.id` unique globally; `source.id` unique within its subject.
- Every `claim.source` resolves to a source in the same subject.
- Every source has `archive.url` (Wayback-only at MVP).
- `tier ∈ {1,2,3,4}`.
- Every subject has ≥1 P31 claim; P31 values all appear on the allowlist.
- Footnote resolution: every `[^label]: body` resolves to exactly one source across the article's `subjects[]`. Ambiguity or zero-match fails with a specific error.
- No orphan subjects (subject referenced in `articles[].subjects` must exist).

### Reviewer MVP

The reviewer runs as a GitHub Action on PRs to `frisco-wiki`. **Advisory** only — posts a comment, does not block merge.

```
PR opened/synced
  → Action checks out frisco-wiki@PR-head
  → collects: changed articles, their referenced subjects, citations' archive URLs
  → runs three checks via Claude API (each a separate structured call)
  → composes a single PR comment with findings grouped by check
```

**Coverage** (cheap, no network). Reads article prose + referenced subjects' claims. Flags unbacked or overreaching prose.

**Support** (fetches archive URLs). For each citation, pulls the Wayback snapshot text and checks whether it supports the cited prose. Runs only on assertions that passed coverage.

**NPOV flags** (prose analysis). Fed by `docs/editorial-principles.md` (extracted from the existing plan doc's §2). Flags loaded language, unattributed advocacy, synthetic consensus, failure to name disputants.

Determinism is a design goal: `temperature: 0`, pinned model, content-hash-keyed cache in the Action's artifact cache. No per-PR budget cap is enforced in code at MVP — spend is tracked externally via Anthropic console.

At MVP, only trusted contributors PR to `frisco-wiki`, so the Action uses `pull_request` with API keys in repo secrets. Migrating to `pull_request_target` with trusted-workflow checkout is a documented later task, gated on the arrival of external contributors.

### Authoring tooling

Two small scripts keep the authoring loop humane:

- `scripts/new-subject.ts --qid Q99524088 --slug jackie-fielder` — scaffolds a subject file from Wikidata, fetching label, description, and a handful of P-values. One-time fetch; the resulting YAML is authoritative post-scaffold. Not Wikidata-bound at build time.
- `scripts/ensure-archived.ts` — for each source URL in a PR diff, calls Wayback's Save Page Now API (authenticated via `ARCHIVE_ORG_S3_*` env vars when provided; anonymous otherwise), polls for capture completion, writes the resulting snapshot URL back into the source record. Runs locally pre-commit; not a CI step at MVP.

## Existing Patterns

`friski-code` is greenfield — no prior code to follow. This design establishes the patterns that later phases will extend:

- **Three-repo split** with content separated from code. `frisco-wiki` stays deliberately thin. Enables cleaner contributor-facing surfaces later and portability across forges (Codeberg mirror is planned per `plans/2026-04-19-plan-0.1.md` §3.2).
- **Claims as first-class**, not denormalized in article frontmatter. Subjects own claims; articles are prose views that reference subjects. Editorial policies are derived from the claim graph rather than tagged inline.
- **Deterministic cross-references.** Every pointer (claim→source, footnote→source, article→subject) resolves by exact lookup. No fuzzy matching. This keeps reviewer check targets unambiguous.
- **Shell scripts + thin YAML wrappers** for CI (plan §3.2). Logic lives in portable Node/shell; forge-specific YAML is a thin wrapper. Codeberg/Forgejo migration is cheap when it becomes relevant.

The existing `plans/2026-04-19-plan-0.1.md` document provides the editorial principles (§2), thesis framing (§1), and long-term architectural intent this design materializes. It is superseded by this design plan as the canonical Phase 0 technical reference; its §2 remains normative for editorial policy until extracted into `docs/editorial-principles.md` during implementation (Phase 4).

## Implementation Phases

### Phase 1: Project Setup & Schemas

**Goal:** Initialize `friski-code` as an Astro 5 project with typed content schemas. No rendered content yet.

**Components:**
- `friski-code/package.json` with Astro 5, TypeScript, Zod, js-yaml dependencies
- `friski-code/astro.config.mjs` — Astro configuration
- `friski-code/tsconfig.json` — strict TypeScript
- `friski-code/src/content.config.ts` — Zod schemas for subjects, articles, sources, claims (shapes per Architecture)
- `friski-code/config/allowed-types.yaml` — initial P31 allowlist (Q5 human, plus 2–3 institutional/place types)
- `friski-code/src/lib/subject-graph.ts` — type definitions for the in-memory graph
- `frisco-wiki` repo created empty with README explaining layout
- `.gitmodules` wiring `frisco-wiki` at `src/content/wiki/`

**Dependencies:** None.

**Done when:** `npm install` succeeds; `npm run build` succeeds on an empty content directory; Zod schemas compile; a unit test loading a hand-crafted valid subject YAML against the schema passes.

### Phase 2: Content Loader & Validator

**Goal:** Astro reads, validates, and graphs the content. `validate-content.ts` catches schema and cross-reference violations.

**Components:**
- Custom content-layer loader in `friski-code/src/content.config.ts` that reads from `src/content/wiki/{subjects,articles}/`
- `friski-code/src/lib/subject-graph.ts` — in-memory graph construction: `Subject[]`, `Article[]`, lookup maps; derived predicates (`types`, `is_living_person`, `active_claims(prop)`)
- `friski-code/scripts/validate-content.ts` — runs the validator rules from Architecture
- Fixture content in `friski-code/tests/fixtures/` — known-valid and known-invalid subject/article files to test the validator

**Dependencies:** Phase 1.

**Done when:** Loading the valid fixtures populates the graph correctly. The validator rejects each invalid fixture with a specific error message. Each validator rule listed in Architecture has a passing test and a failing test.

### Phase 3: Page Rendering

**Goal:** The site renders articles, subject pages, and at least one derived index page from the graph.

**Components:**
- `friski-code/src/layouts/BaseLayout.astro` — base layout, minimal styling
- `friski-code/src/pages/[slug].astro` — article renderer; resolves footnotes to publication + date + archive link
- `friski-code/src/pages/subjects/[id].astro` — subject page with claim table and article backlinks
- `friski-code/src/pages/index/current-supervisors.astro` — one concrete query-derived index page, proof of the pattern
- `friski-code/src/components/` — Citation, ClaimsTable, SubjectReference components as needed

**Dependencies:** Phase 2.

**Done when:** `npm run build` against the validator fixtures produces a deployable static site. Article page renders prose with resolved citations. Subject page renders claims and backlinks. Index page renders a filtered result iterating the graph. Lighthouse accessibility score on the three page types is non-embarrassing (>85).

### Phase 4: Authoring Tooling & Editorial Docs

**Goal:** New subjects can be scaffolded from Wikidata. Source URLs can be submitted to Wayback and snapshot URLs written back. Editorial principles and archival procedure are documented in reviewer-consumable form.

**Components:**
- `friski-code/scripts/new-subject.ts` — CLI: `--qid <Q>`, `--slug <slug>`; fetches Wikidata entity; writes `frisco-wiki/subjects/<slug>.yaml` with P31, label, description, seeded properties; validates against Zod schema
- `friski-code/scripts/ensure-archived.ts` — CLI: reads source records; calls Wayback Save Page Now; polls for completion; writes `archive.url` and `archive.method: wayback` back
- `friski-code/docs/archival-procedure.md` — MVP workflow: Wayback-friendly outlet list, how to run `ensure-archived.ts`, what to do when a source can't be archived (MVP answer: pick a different source)
- `friski-code/docs/editorial-principles.md` — principles extracted from `plans/2026-04-19-plan-0.1.md` §2 into a reviewer-consumable form

**Dependencies:** Phase 2 (validator).

**Done when:** Running `new-subject.ts` against a real Wikidata QID produces a schema-valid subject file. Running `ensure-archived.ts` against a Mission Local URL that has never been archived triggers a capture and writes back the resulting snapshot URL. Integration tests cover success and failure paths for both scripts. `editorial-principles.md` and `archival-procedure.md` exist and have been reviewed by Luis.

### Phase 5: Seed Corpus

**Goal:** 2–3 real articles in `frisco-wiki` passing the validator, rendered by the site. Built *using* the Phase 4 tooling, not hand-authored.

**Components:**
- `frisco-wiki/subjects/jackie-fielder.yaml`
- `frisco-wiki/subjects/sf-board-of-supervisors.yaml`
- 1–2 additional subjects from a different category (place, institution, or event) to exercise the schema beyond persons
- `frisco-wiki/articles/jackie-fielder.md`
- 1–2 additional articles corresponding to the additional subjects
- At least one cross-subject reference (article A references a subject that also appears in article B — verifies backlinks)
- At least one subject with ≥3 claims (exercises claim table rendering)
- `friski-code` submodule updated to the seeded `frisco-wiki` commit

**Dependencies:** Phase 3 (renderer), Phase 4 (tooling).

**Done when:** Validator passes on the corpus. `npm run build` renders the site without errors. Article pages, subject pages, and the index page show the seeded content correctly. At least one end-to-end authoring loop (scaffolder → archive → write article → validate → render) is walked by Luis and takes under 30 minutes for a straightforward subject.

### Phase 6: Reviewer MVP

**Goal:** Claim coverage, source support, and NPOV checks run against a PR's changed content and produce a single structured comment. Output judged actionable and honest on real content.

**Components:**
- `friski-code/scripts/reviewer/` — Node runner
  - `index.ts` — entry point invoked by CI; collects PR context, runs checks, composes comment, posts via GitHub API
  - `checks/coverage.ts` — claim-coverage check; reads article prose + referenced subjects' claims; calls Claude API with structured prompt; parses structured response
  - `checks/support.ts` — source-support check; fetches archive URLs; calls Claude API per citation
  - `checks/npov.ts` — NPOV check; calls Claude API on full prose with editorial-principles doc as context
  - `prompts/` — prompt templates as separate files
  - `cache/` — content-hash-keyed cache avoiding re-review of unchanged content
- `friski-code/tests/reviewer-fixtures/` — fixture PRs with known issues: a claim-less assertion, an overreach beyond source, an advocacy-voiced paragraph, a healthy baseline

**Dependencies:** Phase 5 (real content to review against).

**Done when:**
- Running the reviewer against the seed corpus (healthy content) produces findings Luis judges *actionable and honest* — not noise, not flattery. Zero-finding output is acceptable on clean content.
- Running the reviewer against each fixture PR produces at least one correctly-pointed finding for the known issue in that fixture.
- The reviewer exits cleanly on both happy and error paths (API timeout, missing archive URL, malformed response).
- Composed comment is well-formed markdown.

### Phase 7: CI, Deploy & Round-Trip Verification

**Goal:** Validator and reviewer run automatically on PRs; merged content deploys to `frisco.wiki`; the Definition of Done round-trip works end-to-end.

**Components:**
- `friski-code/.github/workflows/validate.yml` — runs `validate-content.ts` on PRs touching code or submodule
- `friski-code/.github/workflows/build.yml` — Astro build check
- `friski-code/.github/workflows/submodule-bump.yml` — on `repository_dispatch` from `frisco-wiki`, opens a PR bumping the submodule
- `frisco-wiki/.github/workflows/validate.yml` — calls validator code from `friski-code` main
- `frisco-wiki/.github/workflows/review.yml` — invokes reviewer from `friski-code` main; Claude API key and (optional) archive.org S3 keys in repo secrets
- `frisco-wiki/.github/workflows/notify-code.yml` — on merge to main, `repository_dispatch` event to `friski-code`
- Netlify site configured to build `friski-code` on merges to main; `frisco.wiki` DNS pointed at Netlify
- `friski-code/README.md` updated with workflow description

**Dependencies:** Phases 1–6.

**Done when:** The full Definition of Done round-trip works on a new (post-seed) subject + article + capture:
1. PR to `frisco-wiki` runs validator and reviewer; both produce actionable output.
2. Merging the PR triggers an automated submodule-bump PR against `friski-code`.
3. Merging the bump PR triggers a Netlify build.
4. `frisco.wiki` renders the new article, subject page, and any affected index pages.
5. Total wall-clock from PR-opened to content-live is under ~15 minutes (directional, not a hard SLO).
6. Luis can add one more subject + article afterward without consulting implementation docs.

## Additional Considerations

**Editorial scope of demonstration content.** Seed-corpus subjects should avoid unresolved editorial risk at the demonstration stage. Use Jackie Fielder rather than Connie Chan; pick institutions and places with broad neutral coverage. The wiki's eventual content coverage is not constrained by this choice — it applies only to MVP demonstration artifacts.

**Fork-PR security posture.** The MVP reviewer trusts PR authors because at MVP there is only one author. Before external contributors are invited (Phase 3+ per the existing plan doc), migrate the reviewer action to `pull_request_target` with trusted-workflow checkout of reviewer code from `main`, so untrusted PR code cannot read the Claude API key. Named explicitly here so it isn't forgotten when contributors arrive.

**Paywalled sources explicitly out of scope.** MVP restricts source outlets to Wayback-friendly publications (Mission Local, KQED, SF Public Press, 48 Hills, Bay City News, government records). Paywalled outlets (SF Chronicle, SF Standard) are deferred with full thinking preserved in `plans/deferred/archival-and-captures.md`. Adding paywalled sources requires Phase 1 work (WARC capture workflow, `frisco-archives` repo, browser-assisted capture tooling).

**Cost management.** No per-PR budget cap is enforced in reviewer code at MVP. Spend is monitored via Anthropic console and billing alerts. `temperature: 0` and content-hash caching should keep repeat runs cheap. If spend surprises happen, a budget cap re-enters scope.

**Superseding the existing plan.** Merging this design plan obsoletes `plans/2026-04-19-plan-0.1.md` as the normative technical reference. Future sessions should treat this design plan as authoritative; the older document remains in-repo for narrative/historical context but its technical architecture is superseded. The `CLAUDE.md` pointer is updated in the same commit.

## Deferred features (post-MVP)

**Wiki-style `[[ ]]` links and redlinks.** Authors writing articles would benefit from inline links to other subjects written as `[[jackie-fielder]]` or `[[jackie-fielder|Fielder]]`, resolved at build time against the subject graph. Links to existing subjects render as normal `<a href="/subjects/<slug>">`; links to missing subjects render as Wikipedia-style "redlinks" — distinct CSS class, same `href` that 404s until the subject lands, signaling to authors that the target subject is an open writing task. Implementation is a remark plugin paralleling the Phase 3 `rehype-rewire-footnotes` pattern. Design decisions to settle when the work starts:

- **Relationship to `subjects[]` frontmatter.** Preferred: `subjects[]` remains the *semantic* assertion ("this article is about these entities"; drives `articlesReferencing` backlinks and graph membership), while `[[ ]]` remains a *presentation* assertion ("I'm referencing this subject in prose"). Keep them distinct so editorial intent stays legible; the two lists overlap heavily in practice but don't have to match exactly.
- **Graph implications.** Wiki-links could optionally be scanned by the graph builder as implicit references for backlink purposes — but doing so conflates authoring convenience with editorial intent. Default: no. Revisit if authors find themselves duplicating heavily.
- **Redlink destination.** Two options: `href="/subjects/<slug>"` that 404s (simple, signals the gap), or `href="/contribute?subject=<slug>"` that points at a scaffolding flow (more helpful, more to build). MVP-style redlink should just be the 404 variant.
